3.  **Tipos de Pruebas:**
    *   Descripción detallada de los diferentes tipos de pruebas que se ejecutarán:
        *   **Pruebas Unitarias:**
            *   Propósito: Verificar que unidades individuales de código (funciones, métodos, componentes pequeños) funcionan correctamente de forma aislada.
            *   Aplicación: En módulos clave del backend (lógica de negocio, validación de datos, interacción básica con BD/APIs externas) y componentes reutilizables del frontend.
        *   **Pruebas de Integración:**
            *   Propósito: Verificar la correcta interacción y comunicación entre diferentes módulos o servicios (ej. Frontend consumiendo API del Backend, Backend interactuando con Cloud SQL, Backend subiendo/descargando a Cloud Storage, Backend llamando a la API de IA).
            *   Enfoque: Asegurar que los datos se transfieren correctamente y que las interfaces entre los componentes funcionan según lo esperado.
        *   **Pruebas Funcionales (End-to-End):**
            *   Propósito: Verificar que cada funcionalidad completa del Dashboard, desde la interacción del usuario en el frontend hasta la acción en el backend y base de datos, funciona según los requisitos.
            *   Enfoque: Probar cada flujo de usuario clave detallado en el Documento 1.
        *   **Pruebas de UI/Usabilidad:**
            *   Propósito: Verificar que la interfaz de usuario es intuitiva, fácil de navegar, visualmente consistente (según la guía de estilo Glassmorphism Oscuro) y que los elementos interactivos se comportan como se espera.
            *   Enfoque: Probar la experiencia del usuario al interactuar con los componentes (botones, formularios, tablas, menús, etc.) y la navegación entre vistas.
        *   **Pruebas de Seguridad:**
            *   Propósito: Identificar vulnerabilidades y asegurar que la aplicación protege los datos sensibles y controla el acceso adecuadamente.
            *   Enfoque: Probar la autenticación (login con credenciales válidas/inválidas), autorización (verificar que cada rol solo tiene acceso a las funcionalidades y datos permitidos), validación de entrada para prevenir ataques (ej. inyección SQL, XSS), seguridad en el manejo de archivos subidos, seguridad de las APIs.
        *   **Pruebas de Rendimiento:**
            *   Propósito: Medir la velocidad, capacidad de respuesta y estabilidad de la aplicación bajo diferentes cargas.
            *   Enfoque: Probar el tiempo que tarda en cargar la tabla de datos con un volumen grande de registros, el tiempo de respuesta de las APIs críticas (ej. búsqueda de pacientes, obtención de historial), el rendimiento de la interfaz al navegar o interactuar con componentes complejos, y el impacto del uso del Visor de Archivos Visuales. Considerar el impacto del volumen proyectado de 2TB de datos (principalmente en Cloud Storage) en el rendimiento general.
        *   **Pruebas de Regresión:**
            *   Propósito: Asegurar que los cambios o nuevas funcionalidades no han introducido errores en partes existentes del sistema.
            *   Cuándo se ejecutarán: Después de cada nueva implementación significativa o corrección de errores.

5.  **Pruebas Específicas Importantes (Ejemplos de Enfoque):**
    *   Enfoque en probar a fondo las funcionalidades críticas como:
        *   Ciclo completo de gestión de Pacientes (Crear, Ver, Editar, Archivar, Eliminar Suave).
        *   Funcionalidades de Búsqueda y Filtrado (probar diferentes criterios, casos borde, búsqueda vacía).
        *   Gestión Documental (Subir diferentes tipos de archivos, ver archivos, eliminar suave).
        *   Gestión de Citas (Crear, Editar, Eliminar, probar con fechas pasadas/futuras).
        *   Todas las interacciones definidas con el Asistente Virtual (probar diferentes comandos/preguntas, verificar respuestas).
        *   Pruebas de Seguridad: Intentar acceder a funcionalidades de Administrador con un usuario Asistente, intentar eliminar un registro con borrado fuerte si solo se permite borrado suave, probar inyecciones simples en campos de entrada.
        *   Pruebas de Roles y Permisos: Crear matriz de permisos y probar que cada rol solo puede ejecutar las acciones permitidas.
        *   Pruebas de Borrado Suave: Verificar en la base de datos que los registros se marcan como inactivos y no se eliminan físicamente.
